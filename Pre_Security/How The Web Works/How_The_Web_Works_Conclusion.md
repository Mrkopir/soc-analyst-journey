## Technical Understanding

### Domain Structure

A domain name consists of several parts:

- **Top-Level Domain (TLD)** – the rightmost part of the domain name (for example `.com`, `.ua`)
- **Second-Level Domain (SLD)** – the main name of the website
- **Subdomain** – a prefix used to organize services or sections of a website (for example `mail.example.com`)

### DNS (Domain Name System)

DNS is responsible for translating domain names into IP addresses.

When a user enters a domain into a browser, the DNS resolution process happens in several steps:

1. The system first checks the **local DNS cache**
2. If the IP is not found, the request is sent to a **recursive DNS server** (usually provided by the ISP)
3. The recursive server queries the **root DNS servers**
4. The root servers direct the request to the appropriate **TLD servers**
5. The **authoritative DNS server** provides the final DNS record with the correct IP address

Common DNS record types include:

- **A Record** – maps a domain to an IPv4 address  
- **AAAA Record** – maps a domain to an IPv6 address  
- **CNAME Record** – an alias for another domain  
- **MX Record** – mail server records  
- **TXT Record** – text information.

### HTTP Requests and Responses

Communication between a browser and a web server happens through **HTTP requests and responses**.

- The **client sends a request** to the server
- The **server processes it and returns a response**

The response may include a webpage, image, or other resources.

### HTTP Methods

HTTP methods define what action the client wants to perform.

Common methods include:

- **GET** – retrieve data from the server
- **POST** – send data to the server
- **PUT** – update an existing resource
- **DELETE** – remove a resource

### HTTP Status Codes

HTTP status codes indicate the result of a request.

Examples include:

- **200 OK** – request successful
- **404 Not Found** – resource does not exist
- **403 Forbidden** – access denied
- **500 Internal Server Error** – server encountered an error

### Headers and Cookies

**HTTP headers** contain additional information about requests and responses such as:

- content type
- authentication data
- caching policies

**Cookies** are small pieces of data stored in the user's browser. They are commonly used for:

- session management
- authentication
- user preferences

### Web Technologies

The course also covered basic web technologies.

- **HTML** – the standard language used to structure web pages
- **JavaScript** – used to create dynamic and interactive web functionality

Websites often rely on **backend scripting languages** (such as Python, PHP, Node.js, etc.) to generate dynamic content and interact with databases.

### Web Infrastructure

Modern websites rely on multiple infrastructure components:

- **Load Balancers** – distribute traffic across multiple servers
- **Content Delivery Networks (CDN)** – cache content closer to users to improve performance
- **Databases** – store application data
- **Virtual Hosts** – allow multiple websites to run on the same server
- **Static Content** – files served directly (HTML, CSS, images)
- **Dynamic Content** – generated by the server using backend logic

## Security Perspective

Understanding how the web works is important for identifying potential security risks.

Web applications may be vulnerable to attacks such as:

- **HTML Injection**
- **Cross-Site Scripting (XSS)**
- **Sensitive Data Exposure**

For example:

- Poor input validation may allow attackers to inject malicious HTML into web pages.
- Sensitive information such as passwords or API keys may be exposed if encryption is not properly implemented.
- Using **HTTP instead of HTTPS** can expose communication to **Man-in-the-Middle (MITM)** attacks.

Security mechanisms used to protect web applications include:

- **HTTPS encryption (TLS)**
- **Web Application Firewalls (WAF)**
- **secure DNS configuration**
- **proper input validation**

These protections help reduce the risk of exploitation and improve the overall security of web services.
